\chapter{Despliegues}
\label{ch:despliegues}

SIPREH se despliega siguiendo la misma separación que define su arquitectura: el frontend y el backend viven en plataformas distintas y se comunican únicamente a través de la API HTTP. En la práctica esto significa dos despliegues independientes que pueden actualizarse por separado:

\begin{itemize}
    \item El \textbf{frontend} se publica como sitio estático en \textbf{GitHub Pages}.
    \item El \textbf{backend} corre como servicio web en \textbf{Railway}, acompañado de la base de datos relacional y del bucket de objetos en Cloudflare R2.
\end{itemize}

Este capítulo describe cómo funciona cada despliegue en producción y qué se necesita para reproducirlo.

\section{Despliegue frontend}
\label{sec:deploy_frontend}

El frontend es una aplicación \textbf{Next.js 16} que, para producción, se compila como \textbf{exportación estática}. En lugar de un servidor Node.js permanente, el resultado es una carpeta \texttt{out/} con HTML, CSS y JavaScript que cualquier hosting de archivos estáticos puede servir. GitHub Pages cumple ese rol sin costo adicional.

\subsection*{Cómo se publica}

El despliegue está automatizado con GitHub Actions. El flujo vive en \texttt{.github/workflows/deploy.yml} y se dispara con cada \textit{push} a la rama \texttt{main}. No hay que ejecutar nada a mano: basta con fusionar a \texttt{main}.

El flujo realiza dos trabajos encadenados:

\begin{enumerate}
    \item \textbf{Build.} Instala las dependencias del frontend con \texttt{npm ci}, compila la exportación estática con \texttt{npm run build} y sube la carpeta \texttt{out/} como artefacto de Pages.
    \item \textbf{Deploy.} Publica ese artefacto en el entorno \texttt{github-pages}, que queda disponible en la URL del proyecto.
\end{enumerate}

\subsection*{La pieza clave: a qué backend apunta el sitio}

Como el sitio es estático, la dirección del backend se ``hornea'' en el código \textbf{durante el build}, no en tiempo de ejecución. Por eso la variable que la define debe estar disponible en el momento de compilar:

\begin{verbatim}
NEXT_PUBLIC_API_URL=https://<tu-backend>.up.railway.app/api/v1
\end{verbatim}

En el flujo de GitHub Actions, este valor se inyecta desde una \textit{variable de repositorio} llamada \texttt{RENDER\_BACKEND\_URL}. Para cambiar de backend (por ejemplo, al migrar de entorno), se edita esa variable en \texttt{Settings $\to$ Secrets and variables $\to$ Actions} del repositorio y se vuelve a desplegar.

\begin{remark}
Como la URL del backend queda fijada en el build, un cambio en \texttt{RENDER\_BACKEND\_URL} solo surte efecto tras un nuevo despliegue. Volver a ejecutar el flujo (o hacer un \textit{push} a \texttt{main}) regenera el sitio con la nueva dirección.
\end{remark}

\subsection*{El detalle del \texttt{basePath}}

Un sitio de GitHub Pages de proyecto no se sirve en la raíz del dominio, sino bajo una subruta con el nombre del repositorio (por ejemplo, \texttt{https://<organización>.github.io/SIPREH/}). Si la aplicación asumiera que vive en la raíz, las imágenes y los enlaces internos se romperían.

Para evitarlo, \texttt{next.config.mjs} detecta la variable \texttt{GITHUB\_ACTIONS} (que GitHub define automáticamente en cada ejecución) y, solo en ese caso, configura \texttt{basePath} y \texttt{assetPrefix} con el nombre del repositorio. En desarrollo local esa variable no existe, así que la aplicación funciona en \texttt{localhost:3000} sin prefijo y sin cambios.

\begin{example}
Comportamiento según el entorno, controlado por una sola condición en \texttt{next.config.mjs}:
\begin{verbatim}
# Local (GITHUB_ACTIONS no definida)
http://localhost:3000/

# GitHub Pages (GITHUB_ACTIONS=true)
https://<organización>.github.io/SIPREH/
\end{verbatim}
\end{example}

\section{Despliegue backend}
\label{sec:deploy_backend}

El backend es una aplicación \textbf{FastAPI + Uvicorn} que se despliega en \textbf{Railway}. Railway construye el servicio directamente desde el repositorio y lo mantiene en línea, exponiéndolo en una URL pública del tipo \texttt{https://<servicio>.up.railway.app}.

\subsection*{Modo producción}

El punto de entrada \texttt{run.py} decide automáticamente si está en producción. Cuando detecta la variable \texttt{RAILWAY\_ENVIRONMENT} (que Railway inyecta sola) o \texttt{RENDER}, o cuando \texttt{DEBUG} no está definida, desactiva la recarga automática de código y escucha en el puerto que la plataforma indique mediante la variable \texttt{PORT}. En desarrollo local, en cambio, la recarga queda activa para agilizar el trabajo.

\begin{remark}
No es necesario configurar el puerto manualmente en Railway: la plataforma asigna \texttt{PORT} y \texttt{run.py} la respeta. Tampoco se necesita un \texttt{Procfile} ni un servidor de procesos externo; Uvicorn es el servidor de la aplicación.
\end{remark}

\subsection*{Variables de entorno}

Toda la configuración se carga desde variables de entorno (en local, desde un archivo \texttt{.env}; en Railway, desde el panel del servicio). La gran mayoría son opcionales y tienen valores por defecto razonables: el backend arranca aunque falten R2, Redis o Groq, degradándose con elegancia. Las relevantes para un despliegue completo son:

\begin{verbatim}
# Base de datos relacional (metadatos, usuarios, registro de archivos)
DATABASE_URL=postgresql://user:password@host:5432/sipreh

# Seguridad
SECRET_KEY=<clave_secreta_larga_y_aleatoria>
BACKEND_CORS_ORIGINS=["https://<organización>.github.io"]

# Almacenamiento de objetos (Cloudflare R2, compatible con S3)
CLOUD_STORAGE_PROVIDER=cloudflare-r2
CLOUD_STORAGE_ENDPOINT=https://<account_id>.r2.cloudflarestorage.com
CLOUD_STORAGE_ACCOUNT_ID=<account_id>
CLOUD_STORAGE_ACCESS_KEY=<clave_de_acceso>
CLOUD_STORAGE_SECRET_KEY=<clave_secreta>
CLOUD_STORAGE_BUCKET=drought-data
CLOUD_STORAGE_REGION=auto

# Caché de resultados (opcional pero muy recomendable)
REDIS_URL=redis://<host>:6379/0

# Resúmenes con IA (opcional)
GROQ_API_KEY=<clave_groq>

# Usuario administrador inicial
ADMIN_EMAIL=admin@sipreh.co
ADMIN_PASSWORD=<contraseña_inicial>
\end{verbatim}

\begin{remark}
\texttt{BACKEND\_CORS\_ORIGINS} debe incluir el dominio de GitHub Pages desde el que el frontend hace las peticiones; de lo contrario el navegador bloqueará las respuestas por política de CORS. Es el ajuste que más suele olvidarse al conectar ambos despliegues por primera vez.
\end{remark}

\subsection*{Inicialización de la base de datos}

La primera vez que se prepara un entorno, se ejecuta \texttt{python init\_db.py}. Este script crea las tablas, da de alta el usuario administrador definido por \texttt{ADMIN\_EMAIL}/\texttt{ADMIN\_PASSWORD} y normaliza metadatos antiguos. El esquema se genera directamente con SQLAlchemy (\texttt{create\_all}); el proyecto no usa migraciones Alembic, de modo que para un primer despliegue basta con este paso.

\subsection*{Verificación}

Una vez en línea, la API publica su documentación interactiva, útil para comprobar que el servicio responde:

\begin{itemize}
    \item \texttt{/docs} --- Swagger UI (interfaz de prueba interactiva).
    \item \texttt{/redoc} --- ReDoc (documentación de referencia).
    \item \texttt{/api/v1/historical/catalog} --- catálogo de datos disponibles (requiere autenticación).
\end{itemize}

\section{Almacenamiento y base de datos}
\label{sec:deploy_db}

SIPREH separa deliberadamente dos tipos de almacenamiento: la base de datos relacional, que guarda solo metadatos, y el bucket de objetos, que guarda los datos científicos pesados. Cada uno se despliega de forma distinta.

\subsection*{Base de datos relacional}

Contiene los metadatos del sistema: usuarios, el registro de archivos Parquet y las relaciones geográficas entre celdas y zonas. Es liviana, y por eso su despliegue es sencillo:

\begin{itemize}
    \item \textbf{Desarrollo:} SQLite, sin instalación. El archivo se crea solo al iniciar el backend, en la ruta de \texttt{DATABASE\_URL}.
    \item \textbf{Producción:} PostgreSQL, idealmente como servicio gestionado. Railway ofrece PostgreSQL como complemento del mismo proyecto, lo que mantiene backend y base de datos juntos.
\end{itemize}

\begin{example}
Una misma variable cubre ambos entornos:
\begin{verbatim}
# SQLite (desarrollo local)
DATABASE_URL=sqlite:///./droughtmonitor.db

# PostgreSQL (producción, p. ej. en Railway)
DATABASE_URL=postgresql://sipreh_user:password@host:5432/sipreh
\end{verbatim}
\end{example}

\subsection*{Almacenamiento de objetos Cloudflare R2}

El bucket R2 guarda todos los archivos Parquet con los datos hidrometeorológicos. La configuración inicial es:

\begin{enumerate}
    \item Crear una cuenta en \href{https://www.cloudflare.com/}{Cloudflare} y activar R2 desde el panel.
    \item Crear un bucket con el nombre que usará \texttt{CLOUD\_STORAGE\_BUCKET} (por ejemplo, \texttt{drought-data}).
    \item Generar un token de API de R2 con permisos de lectura y escritura sobre ese bucket.
    \item Llevar las credenciales a las variables \texttt{CLOUD\_STORAGE\_*} del backend.
    \item Probar la conexión ejecutando la sincronización desde el panel de administración de SIPREH.
\end{enumerate}

\begin{remark}
A diferencia de Amazon S3, Cloudflare R2 no cobra por egreso (\textit{egress}). Esto encaja con el patrón de acceso de SIPREH, donde los archivos Parquet (de 100 MB a varios GB) se descargan desde el bucket en el primer acceso o tras un nuevo despliegue.
\end{remark}

\begin{definition}
\textbf{Egress} (en almacenamiento en la nube): transferencia de datos desde el proveedor hacia el exterior. Servicios como Amazon S3 o Google Cloud Storage cobran por GB de egreso, un costo que puede crecer rápido en aplicaciones que sirven archivos de datos grandes con frecuencia.
\end{definition}
